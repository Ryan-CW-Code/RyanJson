# Gemini Skill Card

名称：`ryanjson-test-engineering`

## 定位
- 面向 RyanJson 单测的零重复覆盖补强与结构优化；fuzz 仅在明确要求时介入。

## 适用场景
- 扩展单元测试的边界覆盖并去除重复覆盖。
- 修复泄漏、崩溃、断言误用与失败路径缺失。
- 覆盖率报告出现关键分支未触达。

## 输入建议
- 失败证据：报错日志、未覆盖分支、崩溃样本。
- 约束条件：是否禁止新建文件、宏配置、内存限制。
- 验收标准：目标分支、泄漏标准、回归命令。

## 硬约束
- 默认不新建测试文件，优先补到现有分类。
- 新增/改动单测必须零重复覆盖，发现重复即合并或删除。
- 断言语义必须区分 Add/Insert 与 Replace 的失败所有权规则。
- 不用业务 assert 代替可恢复错误断言。
- 语义不明确时按 `example/ -> test/unityTest/ -> test/fuzzer/` 取证。
- 用户要求审查 fuzz 覆盖时，先比较当前工作区与暂存区/稳定基线，再判断是否真的需要手动补齐。
- fuzz 手动补齐只允许最小范围进入 SelfTest；数学上可由运行期命中的路径，优先恢复热路径或扩 corpus/runs。

## 术语口径
- 统一按 `../references/terminology.md`。
- 输出必须显式区分：已验证/推断、可恢复错误/不可恢复错误、失败语义。

## 默认提示词
使用 `$ryanjson-test-engineering`，在现有测试分类中扩展 RyanJson 测试：严格零重复、优先覆盖失败路径和边界分支、明确所有权断言；仅在明确要求时涉及 fuzzer，并先判断覆盖缺口应留在运行期还是最小化放入 SelfTest，再基于 `example -> unityTest -> fuzzer` 取证提供可复现回归证据。

## 输出骨架
1. 失败根因与目标分支（标注已验证/推断）。
2. 测试改动清单（文件 + 函数 + 场景）。
3. 覆盖与稳定性证据（分支触达/泄漏/崩溃）。
4. 回归边界与下一步建议。
